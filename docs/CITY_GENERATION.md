# Генерация и физическая жизнь города

## Источник истины

Каждое поселение хранит `Settlement.layout`. Это постоянный план, а не результат отрисовки. Он создаётся один раз из ключа мира и содержит отдельный `DistrictLayoutPlan` для каждого глобального района.

План района фиксирует:

- тип морфологии: `organic`, `radial`, `linear`, `fortified`, `waterfront`, `terraced`;
- локальный центр и главную ось;
- плотность и масштаб кварталов;
- тип стены;
- отдельный ключ дорожной сети.

UI и локальная карта не придумывают город заново. Они читают сохранённый план, физические здания, поля и стройки.

## Как формируется город

1. Глобальный генератор выбирает поселение, его размер и занятые глобальные клетки.
2. Районы получают роли: центр, жильё, рынок, ремесло, крепость, порт, поля или окраина.
3. Морфология определяется типом поселения, ролью района, рельефом и ключом мира.
4. Здания выбирают район по назначению. Ферма не должна появляться в административном центре при наличии полей, а кожевенная мастерская — в жилом ядре при наличии ремесленной окраины.
5. Локальная позиция здания выбирается морфологическим генератором. Главные улицы резервируются до застройки.
6. После размещения вход каждого здания соединяется с ближайшей дорогой.
7. Новое строительство использует тот же план и продолжает существующую форму города.

Одинаковый ключ мира обязан воспроизводить тот же город. При открытии экрана создания нового мира интерфейс предлагает новый случайный ключ, поэтому последовательные миры больше не совпадают без явного выбора пользователя.

## Распределение жителей

Распорядок принадлежит жителю и содержит физические координаты для каждой части суток.

Работающий житель идёт к своему реальному рабочему месту. Безработный не получает общий fallback на первый рынок. Его дневная активность распределяется между:

- стройками;
- полями и сезонными работами;
- управой;
- рынками и мастерскими;
- тавернами и объявлениями;
- домашней работой;
- улицами своего района.

Если интерьер переполнен, оставшиеся люди разбиваются на физические группы не более восьми человек. Одна клетка не может представлять сотни жителей.

## Расширение

При добавлении нового района `normalizeSettlementLayouts` или городской API создаёт для него отдельный план. Строительные заявки не обходят морфологию: они перебирают подходящие районы, резервируют реальную свободную землю и только затем становятся стройкой.

## Миграция

Схема 28 создаёт планы для старых поселений и заново размещает здания старых миров по новой морфологии. Это сознательное изменение: старый прямоугольный шаблон не сохраняется как вечный технический долг.

## Расширение системы

Новый тип планировки добавляется через:

1. `CityLayoutStyle`;
2. правила выбора в `cityMorphology.ts`;
3. генерацию кандидатов застройки;
4. дорожный профиль в `localMap.ts`;
5. smoke-сценарий с отличающейся физической сигнатурой.

Новые правила не должны размещаться в React-компонентах. Владелец физического плана — доменный слой города.
